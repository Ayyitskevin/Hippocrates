# App icon — the remaining design task

The app icon is real artwork the owner supplies; it is not generated by this
repository. This note specifies exactly how to add it so the boundary controls
stay intact, because an asset catalog is a **new app resource** and the scanner
currently pins `PrivacyInfo.xcprivacy` as the sole app resource.

## What the icon should convey

Offline, private, professional, calm. Avoid literal caduceus/Rx clichés if a
cleaner mark is possible; a simple monogram or a ledger/checkmark motif reads
well at small sizes. One flat design, legible at 40pt. No text in the icon.

## How to add it without breaking the boundary

This is a small, reviewed scanner change, mirroring how `Charts` and the
ShareLink seams were added. Do it in its own PR:

1. Add `Hippocrates/Resources/Assets.xcassets` with an `AppIcon` icon set (the
   single 1024×1024 App Store icon plus the required device sizes, or a single
   universal 1024 entry if targeting the modern single-size format).
2. In the Xcode project, add the asset catalog to the app target's **Resources**
   build phase and set `ASSETCATALOG_COMPILER_APPICON_NAME = AppIcon` in both
   Debug and Release app-target build configurations. Both settings changes must
   fall inside the scanner's allowlisted build-configuration keys — extend that
   allowlist in the same PR if needed.
3. Extend the scanner's **sole-resource rule** from the single
   `PrivacyInfo.xcprivacy` entry to an explicit two-item allowlist
   (`PrivacyInfo.xcprivacy`, `Assets.xcassets`), keeping the privacy-manifest
   contract exactly as-is. Add the asset catalog directory and its contents to
   the sandbox input inventory and the scanner's `expectedBoundaryInputPaths`
   in lockstep, and update the pinned resource-phase count.
4. The privacy manifest must remain the app's **only** other resource, and the
   asset catalog must contain image assets only — no data files, no fonts that
   could carry metadata-reading required-reason APIs. If any timestamp or
   metadata API becomes reachable, add the matching required-reason declaration
   to the privacy manifest (see README).

## Launch screen

The project uses `INFOPLIST_KEY_UILaunchScreen_Generation = YES`, so a plain
launch screen is generated with no extra asset. Leave it unless a branded
launch is wanted; a branded one is another reviewed resource change.

## Why this is a separate PR

Every other resource change in this project has been its own reviewed commit
with the scanner extended in lockstep. The icon is no different, and it needs
real art, so it waits for the owner rather than shipping a placeholder that
would fail App Store review anyway.
